# SRS v0.4 개정 작업을 위한 AI 프롬프트 가이드

## 작업 개요
본 작업은 기존 SRS-v0.3 문서에서 시스템 복잡도를 극증시키는 **외부 API 연동(PG 에스크로, NICE 신용평가, 금융사 RaaS)**을 MVP 스코프에서 제거하고, 이를 **관리자(Admin) 수동 개입 및 DB 기반 더미/배치 처리**로 우회하도록 문서를 전면 수정하는 과정입니다. 

AI에게 전체 SRS 문서를 먼저 학습시킨 후, 아래 Step 1부터 4까지의 프롬프트를 순차적으로 입력하여 수정본을 도출하십시오.

---

### Step 1. 스코프 및 시스템 아키텍처 개정 (Section 1 & 3)

**[AI에게 입력할 프롬프트]**
```text
학습한 SRS 문서의 '1.2 Scope'와 '3. System Context and Interfaces' 섹션을 다음 지침에 따라 수정해 주세요.

1. [1.2.1 In-Scope (MVP Phase 1)] 수정
- IS-01의 '에스크로 결제 및 자금 방출 로직'을 '무통장 입금 기반 Admin 수동 예치/방출 확인 시스템'으로 변경하세요.
- IS-03의 'API 실시간 뷰어'를 '운영팀 사전 DB 업데이트 기반 평판 뷰어'로 변경하세요.
- IS-05의 'RaaS 구독'을 '이메일 수기 견적 요청(더미 폼)'으로 변경하세요.

2. [1.2.3 Constraints] 및 [1.2.4 Assumptions] 수정
- PG사 고액 거래 지원, NICE API 법적 허용 등의 가정을 모두 삭제하세요.
- 대신 "MVP 단계의 모든 에스크로 자금은 지정된 법인 계좌로 수기 입출금되며, 플랫폼은 상태값만 기록한다"는 제약사항을 추가하세요.

3. [3.1 External Systems] 수정
- EXT-01(PG사), EXT-02(NICE), EXT-05(금융파트너)를 표에서 완전 삭제하세요. 
- 임시 우회 전략(Fallback) 내용을 삭제하고, "MVP는 외부 API 통신 없이 Vercel 내부 DB망에서 독립적으로 작동한다"는 아키텍처 원칙을 명시하세요.

수정된 마크다운 결과물만 출력해 주세요.
```

---

### Step 2. 기능 요구사항 전면 수정 (Section 4.1)

**[AI에게 입력할 프롬프트]**
```text
SRS 문서의 '4.1 Functional Requirements' 중 F-01, F-03, F-05를 다음 지침에 맞춰 전면 재작성해 주세요. 복잡한 API 연동 코드가 나오지 않도록 바이브코딩 친화적으로 단순화하는 것이 핵심입니다.

1. [F-01: 안심 에스크로 결제 시스템] (REQ-FUNC-001 ~ 005)
- 요구사항 변경: 수요 기업이 플랫폼 지정 계좌로 계약금을 입금하면, Admin이 확인 후 상태를 'escrow_held'로 수동 변경합니다. 
- 자금 방출 로직 변경: 수요 기업이 검수 승인을 누르면, Admin 대시보드에 '방출 대기' 알림이 뜨고, Admin이 수기 송금 후 상태를 'released'로 변경합니다.
- 타임아웃, 재시도(004) 항목은 삭제하세요.

2. [F-03: SI 파트너 평판 뷰어] (REQ-FUNC-009 ~ 012)
- API 한도 초과 및 폴백 요구사항(011, 012)을 모두 삭제하세요.
- 요구사항 변경: 재무 등급은 Admin이 주기적으로 SI_PARTNER 테이블에 직접 업데이트한 정적 데이터를 불러와 표시합니다. 

3. [F-05: RaaS 구독 계산기] (REQ-FUNC-018 ~ 022)
- 금융 API 호출 및 타임아웃 로직(020, 022)을 모두 삭제하세요.
- 요구사항 변경: RaaS 플랜 선택 시 API를 호출하지 않고, "운영팀에 맞춤 견적 요청하기" 팝업이 뜨며, 신청 시 DB에 리드(Lead) 데이터만 저장하고 Admin에게 알림을 보냅니다.

수정된 F-01, F-03, F-05의 표를 마크다운으로 출력해 주세요.
```

### Step 3. API 목록 및 데이터 모델 구조화 (Section 6.1 & 6.2)

**[AI에게 입력할 프롬프트]**
```text
SRS 문서의 '6.1 API Endpoint List'와 '6.2 Entity & Data Model'을 수정해 주세요.

1. [6.1 API List]
- 외부로 나가는 결제, NICE, 금융 연동 API(API-01, 02, 03, 06)를 삭제하세요.
- 대신 Admin이 상태를 변경할 수 있는 내부 Server Action(`action: updateEscrowStatus`, `action: requestManualQuote`) 등을 추가하세요.

2. [6.2.5 ESCROW_TX 테이블 수정]
- `pg_tx_id` 필드를 삭제하세요.
- 입출금 증빙을 위한 `admin_verified_at` (TIMESTAMP, 승인시간), `admin_memo` (TEXT, 관리자 메모) 필드를 추가하세요.

3. [6.2 ERD / Class Diagram 수정]
- 외부 의존성이 제거된 것에 맞춰 클래스의 메서드(deposit, queryCredit 등)를 Admin 승인 기반의 내부 메서드로 단순화하여 다이어그램 스크립트를 재작성해 주세요.

수정된 마크다운 결과물과 Mermaid 코드를 출력해 주세요.
```

---

### Step 4. 시퀀스 다이어그램 단순화 (Section 6.3)

**[AI에게 입력할 프롬프트]**
```text
SRS 문서의 '6.3 Detailed Interaction Models'의 시퀀스 다이어그램을 다음 지침에 맞춰 수정해 주세요.

1. [6.3.1 에스크로 결제 흐름]
- PG사가 개입하는 분기를 삭제하세요. 
- 사용자 입금 -> 플랫폼 안내 -> Admin 입금 확인 -> Admin 상태 변경(held) -> 자동 보증서 발급으로 흐름을 단순화하세요.
- 방출 역시 검수 승인 -> Admin 방출 확인 -> 상태 변경(released) 흐름으로 수정하세요.

2. [6.3.3 RaaS 비용 비교 흐름] & [6.3.4 신용 조회 흐름]
- 금융 파트너 API 호출 및 NICE API 실시간 조회 시퀀스를 완전히 삭제하세요.
- 내부 DB를 단순 조회하여 뿌려주거나, 수기 견적 폼을 제출하여 Admin에게 전송하는 흐름으로 다이어그램을 다시 그려주세요.

수정된 Mermaid 시퀀스 다이어그램 코드 3가지를 생성해 주세요.
```
