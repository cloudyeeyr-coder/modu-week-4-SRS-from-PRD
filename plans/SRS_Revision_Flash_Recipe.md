# MVP SRS 개정 프롬프트 플랜

## 📌 Phase 1: 핵심 범위 및 시스템 컨텍스트 조정
**목적:** 가장 큰 병목인 외부 API와 자동화 배치를 수동/더미(Mock)화하여 개발 리스크를 제거합니다.

**👉 AI에게 입력할 프롬프트 1:**
> "기존 SRS 문서의 `1.2 Scope`와 `3. System Context`를 다음 기준에 맞춰 수정해줘.
> 1. **에스크로 결제 (F-01) 다운그레이드:** 외부 PG사 에스크로 API 연동을 전면 취소하고, 플랫폼 전용 무통장 입금 계좌 안내 및 Admin 대시보드에서의 '수동 입금 확인/방출 승인' 기능으로 변경해.
> 2. **외부 API 더미화 (F-03, F-05):** NICE 기업 신용정보 API와 금융사 RaaS API 연동을 제거해. 대신 Supabase에 미리 업로드해 둔 정적(Static) 데이터를 Read-Only로 불러와 화면에 렌더링하는 형태로 구조를 단순화해.
> 3. **Vercel Cron 제거:** 뱃지 만료, 검수 자동 전환 등 Cron 기반의 스케줄러 자동화 기능을 모두 삭제해. 이를 Admin 대시보드에 '만료 상태 일괄 업데이트'라는 수동 트리거 버튼(Server Action)을 두는 방식으로 시스템 구조도를 재설계해.
> 위 세 가지를 반영하여 1장과 3장의 관련 표와 다이어그램을 업데이트해서 출력해줘."

## 📌 Phase 2: 기능 요구사항(Functional Requirements) 현실화
**목적:** 39개의 요구사항 중 바이브 코딩 시 오류를 유발할 수 있는 복잡한 예외 처리와 비동기 로직을 걷어냅니다.

**👉 AI에게 입력할 프롬프트 2:**
> "수정된 시스템 컨텍스트를 바탕으로 `4.1 Functional Requirements`를 전면 수정해줘.
> 1. 에스크로 타임아웃, API 재시도, API 한도 소진 폴백(Fallback) 등 외부 연동과 관련된 모든 예외 처리 요구사항(REQ-FUNC-004, 011, 012, 022 등)을 삭제해.
> 2. 상태 전환(Status Transition) 로직을 단순화해. 예를 들어, 기한이 지나면 '자동으로 분쟁 상태로 전환'되는 로직을 삭제하고, '운영팀이 Admin에서 수동으로 분쟁 상태로 변경할 수 있다'로 수정해.
> 3. 알림 발송 기능(카카오톡/SMS)은 MVP 단계에서 복잡도를 높이므로, 우선 플랫폼 내 인앱(In-app) 알림 또는 Next.js에서 직접 발송 가능한 이메일(Resend 등 활용) 단일 채널로 축소해줘.
> 수정된 REQ ID와 Acceptance Criteria를 표 형태로 다시 작성해줘."

## 📌 Phase 3: 데이터 모델(ERD) 및 상세 시퀀스 다이어그램 재설계
**목적:** 백엔드 로직을 극도로 단순화하여 데이터베이스 스키마와 클라이언트 간의 상태 불일치를 예방합니다.

**👉 AI에게 입력할 프롬프트 3:**
> "개정된 요구사항을 바탕으로 `6.2 Entity & Data Model`과 `6.3 Detailed Interaction Models`를 수정해줘.
> 1. `ESCROW_TX` 테이블에서 `pg_tx_id` 및 타임아웃/실패 횟수 관련 컬럼을 삭제하고, `admin_approved_at` (관리자 확인 시간) 컬럼을 추가해.
> 2. `SI_PARTNER` 테이블의 재무 등급 갱신 일시(`financial_grade_updated_at`) 등 캐시 관련 컬럼을 삭제하고, 정적 데이터임을 명시해.
> 3. 시퀀스 다이어그램에서 PG사, NICE평가정보, Vercel Cron 엑터를 모두 제거해. 대신 'Admin 운영팀'이 직접 개입하여 Server Action을 호출하고 DB 상태를 업데이트하는 흐름으로 시퀀스 다이어그램(Mermaid)을 재작성해줘."

---

## 📌 Phase 4: 검증 계획 및 비기능 요구사항 동기화
**목적:** 축소된 기능에 맞춰 시스템 성능 목표와 테스트 계획을 정렬합니다.

**👉 AI에게 입력할 프롬프트 4:**
> "마지막으로 `4.2 Non-Functional Requirements`와 `7. Verification and Validation`을 현재 수정된 스펙에 맞춰 다듬어줘.
> 1. PG사 타임아웃(REQ-NF-003) 등 외부 API 성능 지표를 삭제해.
> 2. AS 24시간 출동 등 강제 SLA 모니터링 자동화 로직이 제거되었으므로, 관련 운영 KPI 달성은 플랫폼 시스템의 기능이 아닌 '운영팀의 수동 관제(Ops)' 영역으로 명시해.
> 3. 리스크 레지스트리(Risk Registry)에서 외부 API 장애 리스크를 삭제하고, '운영팀의 수동 업무 과부하(Admin 병목)'를 새로운 리스크로 추가한 뒤, Phase 2에서 자동화로 해결한다는 완화 전략을 작성해줘."
