## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 타겟 데이터 확인: `Contract.status`가 최소 `release_pending` 혹은 `completed` 인 건만 워런티 발급 가능.
- [ ] `Warranty` 모델 생성:
  - `expiresAt`: 현재일부터 +N년 (기본 1년 등 정책 명시)
  - `documentUrl`: 우선 발급된 임시 PDF URL을 모킹(Mocking) 하거나 파일 서버 통신 결과를 기입할 영역 구축
- [ ] 만약 같은 계약에 이미 워런티가 있다면 이전 것을 보존하거나 `update` 하지 말고 즉시 `에러`를 던지는 중복 방어 락.

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 동시 워런티 발급 차단
- Given: 계약 상태가 `completed` 인 방출 세트
- When: 어드민이 발급 버튼을 두 번 눌러서 2번의 서버 액션이 진입함
- Then: 이 FC 계층이 쿼리를 날려 기존 `warrantyId`를 체스한 뒤 발급 로직을 중단 하여 데이터의 1대1 제약(DB-011)을 지킨다.
