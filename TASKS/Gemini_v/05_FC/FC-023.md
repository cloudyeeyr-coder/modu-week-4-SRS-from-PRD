## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 알림 발신(Send) 통합 함수 모듈라이징 (API-025 DTO 대응)
- [ ] 통신 브릿지: 카카오톡 비즈메시지 API (또는 Alimtalk 모듈) 호출.
- [ ] 에러 핸들링: 만약 카카오 API에서 `500` 을 내려주거나 `Timeout` 이 나면 -> `try-catch`로 묶어서 `SMS` 벤더 라우팅.
- [ ] 최후 방어선 DB Fallback: SMS 마저 터지거나 연락처가 비정상이면, 생성해 둔 `Notification` (DB-014) DB 테이블에 `channel: 'in_app'`, `status: 'unread'` 로 Insert 하여 사용자가 웹페이지 종 모양(bell) 에서라도 볼 수 있게 영속화 한다.

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 에러 체인 (Error Chaining) 우회 성능
- Given: 카카오 API 서버의 일시 다운 (의도적인 Error 모킹)
- When: 시스템에서 이벤트 발송을 지시함
- Then: 이 FC 레이어는 파괴되지 않고 부드럽게 Exception을 잡아낸 뒤 DB Insert로 우회하여 해당 내역을 데이터 레이어에 남긴다 (Fall-Through).

## :gear: Technical & Non-Functional Constraints
- 구조 제약: Notification DB 조차 터졌을 경우에는 메인 로직의 파괴를 무조건 막기 위해 `console.error` 로 로깅시스템에 넘기고 해당 Promise를 조용히 `Resolve` 시키는 단호함이 필요. (알람 하나 꼬였다고 에스크로 돈 방출 로직이 롤백되는 재난 방지).
