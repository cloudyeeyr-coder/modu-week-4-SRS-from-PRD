## :white_check_mark: Task Breakdown (실행 계획)
- [ ] `modelCode` 기반 마스터 데이터 Fetch (마스터 DB에 가격 `basePrice` 등을 쿼리)
- [ ] 입력 오류 검출 파이프라인: 만약 `modelCode`가 DB에 없는 모델이거나, 렌탈 불가능 속성이면 조기 반환(`RAAS_MODEL_NOT_FOUND`).
- [ ] 3가지 계산식 알고리즘 탑재:
  - 1. 일시불(Opex): `basePrice` * 수량 + (기본 유지보수비)
  - 2. 운용리스: ( `basePrice` * 이자율 계수 등 정책 ) / 개월 수
  - 3. RaaS(구독): ( `basePrice` + 플랫폼 보험료 + S/W 요금 등 정책 조합 ) / 개월 수
  - *(참조: 상세 계산식은 회사 재무 파트에 따라 지속 업데이트 되므로 별도 모듈러 분리를 강제함)*
- [ ] 결과값을 API-020의 `RaasCalculationResponse` DTO 포맷으로 조립하여 안전하게 반환.

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 마스터 데이터 참조의 안정성
- Given: 클라이언트가 "오타난 모델이름" 을 폼으로 전송함
- When: 서버가 이 엔진 코드를 실행하려 함
- Then: Null Pointer 나 연산 NaN 에러를 터뜨리는 대신, 조기 단계에서 `DB-016` 쿼리에 잡히지 않음을 인지하고 적법한 "모델 확인 불가" 에러 코드를 사출.

Scenario 2: 연산 공식의 분리(Decoupling) 모델
- Given: 재무팀이 RaaS 마진율을 15%에서 18%로 급거 변경 지시함
- When: 엔지니어가 코드를 뜯어 고침
- Then: 이 FC 레이어 내 핵심 계산부는 상수(Constant)나 별도 클래스로 떨어져 있어서 단 한 줄 수정만으로 모든 연산 체계가 바로 퍼블리싱 되게 유지보수성 좋게 결합되어 있다.
