## :white_check_mark: Task Breakdown (실행 계획)
- [ ] NextAuth v5(혹은 Supabase Auth) 프로바이더(Credentials) 초기화 모듈 세팅.
- [ ] `authorize` 콜백 구현: 로그인 시 입력받은 이메일패스워드를 `bcrypt.compare` 로 DB 조회 및 매칭 검증.
- [ ] `jwt` 콜백: 검증 성공 시 토큰에 플랫폼 역할(`role: 'buyer' | 'si' | 'admin' 등`) 주입.
- [ ] `session` 콜백: 클라이언트가 꺼내 쓸 수 있도록 토큰 데이터를 `session.user` 등에 맵핑.
- [ ] `middleware.ts` 작성: 경로 기반 라우트 제어 (`/admin` 경로는 role='admin' 만, `/dashboard/buyer` 는 'buyer' 만 접근).

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 라우트 가드 블로킹 테스트
- Given: `buyer` 권한만을 가지고 로그인 한 수요기업 인증 세션
- When: 어드민 페이지(`/admin/users`) 또는 SI 파트너 대시보드 경루를 직링(Direct Link)으로 치고 들어감
- Then: 미들웨어 파이프라인이 Role Mismatch를 감지하고 즉각 루트(`/`) 또는 403 Forbidden 페이지로 Redirect 튕겨냅니다.

Scenario 2: 토큰 무결성 서명
- Given: JWT 쿠키 기반의 로그인 상태
- When: 사용자가 개발자 도구를 열어 Role 정보를 수동 임의 조작함
- Then: 서버 측 서명 값 불일치로 세션이 파기되거나(NextAuth 내장 기능 검증) 조작 데이터를 무시하여 인가 위반 범죄를 원천 예방한다.

## :gear: Technical & Non-Functional Constraints
- 성능 검토: 미들웨어는 Edge Runtime 상에서 구동되어야 하므로 (`bcrypt`나 Node.js 내장 모듈 사용 제약), 비밀번호 검증은 Node 환경인 API 내부에서 수행하고 미들웨어는 "가벼운 JWT 포장 까보기" 역할만 전담해야 함.
