## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 응답 결정 DTO 정의: 파라미터 `proposalId` 와 `decision` (`accept` | `reject`)
- [ ] 맵핑 규칙(주석 명세 가이드): '수락(accept)' 인 경우, 상태 전이 뿐만 아니라 내부적으로 `issueBadge` 액션을 일으켜 SI 파트너의 프로필 가치를 올려야 한다는 비즈니스 연계 룰을 인터페이스 파일에 기록.
- [ ] Response Payload: `{ success, finalState, badgeTriggered: boolean }` 와 같이 뱃지 발급 유무를 컴포넌트에 알려주는 데이터 디자인.
- [ ] 에러 코드 목록 작성 (`PROPOSAL_NOT_FOUND`, `PROPOSAL_EXPIRED_OR_CLOSED`)

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 허용되지 않은 상호작용
- Given: 이미 거절 처리되었거나(`rejected`), 기한이 만료된(`expired`) 제안의 `proposalId`
- When: 사용자가 화면 상 잔존한 버튼으로 액션을 찌름
- Then: 이 명세에 따라 구성된 Zod 검증, 혹은 이후의 Context 검증 로직에서 유효하지 않은 액션 오류(`PROPOSAL_EXPIRED_OR_CLOSED`)를 던저 블로킹(Blocking) 시킨다.

Scenario 2: 정상 수락 포맷팅
- Given: 최신 상태의 미응답 제안서
- When: 액션을 `accept` 로 실어서 보냄
- Then: 리턴 타입 안에 `badgeTriggered: true` 속성이 포함된 채로 반환되어 시스템 파이프라인에서 수락→뱃지 생성이 이어졌음을 증명한다.
