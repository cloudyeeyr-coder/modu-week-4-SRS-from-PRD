## :dart: Summary
- 기능명: [API-025] 다채널(카카오/SMS/이메일/웹 기반) 통합 알림 발송 (`POST /api/notifications/send`) Route Handler DTO
- 목적: 카카오 알림톡(우선) → SMS 폴백(Fallback) → 이메일 전송 체인을 지원하는 통일된 단일 호출 인터페이스 규격을 설계한다. 외부 서비스 장애 발생 대비 우회(In-app) 조건까지 흡수한 기반 설계.

## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 단일화된 Request Body DTO 구성
  - `recipientId` (수신대상 UID, 연락처/메일 조회를 위해 사용)
  - `notificationType` (템플릿 매핑을 위한 Enum 등 식별자: ex `escrow`, `as`)
  - `payload` (동적 변수 치환용: 이름, 기간, 금액 등)
  - `priorityChannel` (옵션 사항, 발송 채널 강제: `kakao`, `sms`, `email`, `in_app`)
- [ ] (규칙 명세) 이 API 모델 호출 시, 만약 외부 API(카카오 등)가 시간 초과 나면 자동으로 내부 DB (API-014 구축)에 저장/FallBack 된다는 논리 연계 주석 기입
- [ ] 응답 반환 코드 설계 (성공: 발송 채널 이력 반환 `{ success: true, deliveredVia: 'sms' }`)
- [ ] 에러 코드 목록 작성 (`NOTIFICATION_FAIL_ALL`, `USER_CONTACT_MISSING`)

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 동적 템플릿 변수 맵핑 
- Given: `{ type: "proposal", payload: { partnerName: "SI_A", deadline: "5days" } }` 를 담은 발송 페이로드
- When: 서버로 POST 요청함
- Then: 이 명세를 구현하는 핸들러는 Zod 통과 후, 내부 시스템에 정의된 템플릿 코드에 알맞은 파라미터를 제공해야 한다. (Type 안전성 확보)

Scenario 2: 장애 회피 응답 명세 (Fallback)
- Given: 카카오/SMS 연동이 모두 불가능한 상황적 환경 (모의)
- When: 시스템이 알림을 보내달라고 이 API를 호출함
- Then: API는 500에러를 반환해 시스템 전체를 파괴하는 것이 아니라, `{ deliveredVia: 'in_app' }` 형태를 반환하여 안전하게 DB 처리로 우회했음을 DTO 구조화해야 한다.

## :gear: Technical & Non-Functional Constraints
- 성능 검토: 내부 연동망에서 대량 배치(Batch)로 이 URL을 호출할 수 있으므로, 비동기 큐 전환을 염두에 둔 넉넉한 Timeout 혹은 `{ status: "queued" }` 반환 여부를 DTO 명세 시 고려할 것.
