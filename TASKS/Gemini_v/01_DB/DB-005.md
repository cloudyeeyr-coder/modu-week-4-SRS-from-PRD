### Phase B: 관계선(Relations) 및 정밀 데이터 타입 작성
- [ ] `siPartnerId`, `buyerCompanyId` 이중 외래키(Foreign Key) 지정 및 참조 필드 구성
- [ ] 금전 수치 오류 방지를 위한 정밀 `Decimal` 타입 매핑 (`@db.Decimal(15,2)` 사용, 로컬 호환 고려)
  ```prisma
  model Contract {
    id                 String    @id @default(cuid())
    buyerCompanyId     String
    siPartnerId        String
    totalAmount        Decimal   @db.Decimal(15,2)  // 호환성: SQLite는 Real로 변환됨 유의
    status             String    @default("pending") // Enum 호환 Fallback
    inspectionDeadline DateTime? @db.Date
    createdAt          DateTime  @default(now())
    updatedAt          DateTime  @updatedAt

    buyerCompany       BuyerCompany @relation(fields: [buyerCompanyId], references: [id])
    siPartner          SiPartner    @relation(fields: [siPartnerId], references: [id])
    
    // Relations (Template)
    // escrowTx EscrowTx?
    // asTickets AsTicket[]
  }
  ```

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: 이중 Foreign Key 구조 생성 무결성 확인**
- Given: `BuyerCompany`와 `SiPartner` 두 상위 모델이 기 존재함
- When: 마이그레이션 진행
- Then: DB 스키마에 `Contract` 테이블이 삽입되며, DBMS 레벨 제약 조건(Constraints)으로 `buyerCompanyId` 및 `siPartnerId` FK 제약이 완벽히 바인딩된다.

**Scenario 2: Enum 전이 및 기본값(Default) 검증**
- Given: 에스크로 상태를 감시하는 시스템
- When: 어플리케이션에서 상태 속성에 데이터를 보내지 않고 새 레코드를 생성 Insert 함
- Then: DB 레벨에서 테이블 기본 속성(Default) 기능이 발동하여 `pending` 상태로 초기화된 레코드가 성공적으로 기록되어야 한다.

## :gear: Technical & Non-Functional Constraints
- **정밀도 리스크 통제:** `totalAmount` 필드는 고액의 B2B 거래 트랜잭션이므로 PostgreSQL 배포를 겨냥하여 `Decimal(15,2)` 강제 적용이 원칙이다. 단, Prisma 로컬 SQLite 환경 모드로 실행될 경우 묵시적으로 Decimal 지원 한계에 따라 Real로 동작하는 Prisma 구조적인 특성을 명확히 인지하고 이후 서비스 계층 코드에 문서화해야 한다.
