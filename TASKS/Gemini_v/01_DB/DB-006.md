### Phase B: Prisma 모델 정의
- [ ] `schema.prisma`에 `EscrowTx` 모델 작성 및 `contractId` 연관관계 설정
  ```prisma
  model EscrowTx {
    id              String    @id @default(cuid())
    contractId      String    @unique
    amount          Decimal   @db.Decimal(15,2)
    state           String    @default("held") // Enum Fallback: 'held', 'released', 'refunded'
    heldAt          DateTime?
    releasedAt      DateTime?
    refundedAt      DateTime?
    adminVerifiedAt DateTime?
    adminMemo       String?   // 관리자 입금자/메모 보존 (5년 정책)
    createdAt       DateTime  @default(now())

    contract Contract @relation(fields: [contractId], references: [id])
  }
  ```

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: 모델 간 참조 및 유니크 제약 1:1 관계 검증**
- Given: 계약(`Contract`) 테이블에 ID가 기생성됨
- When: 마이그레이션 적용 후 하나의 `Contract` 레코드에 대해 두 개의 `EscrowTx`를 Insert 시도함
- Then: `@unique` 제약에 걸려 1:1 무결성이 증명되며 DB 오류 반환(Exit Code != 0).

## :gear: Technical & Non-Functional Constraints
- **전자금융거래 보존:** `adminMemo`, `adminVerifiedAt`, `amount`는 소프트 삭제(Soft Delete)가 적용되거나 레코드가 물리적으로 삭제되지 않게 아카이빙(Archive) 아키텍처 관점에서 참조되어야 한다. 이 시점의 DB 설계에서는 1:1 Hard Delete 종속인 `onDelete: Cascade`의 사용을 금지한다.
