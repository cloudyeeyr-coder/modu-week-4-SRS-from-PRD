- [ ] 상태 전이 Update:

  - 수락 시: `status` = `accepted`. 트랜잭션 진행...

- [ ] 제조사에게 "수락(또는 거절) 되었습니다" 알림 큐 배송(`notification`).

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 데드라인 초과 후 응답 방어
- Given: 기한`deadline`이 지난 모의 데이터 레코드
- When: SI 파트너가 뒷북으로 수락 액션을 누름
- Then: FC 내부의 선행 룰 로직에 의해 시스템은 "수락 기한이 지났습니다"(`PROPOSAL_EXPIRED`) 라는 에러를 반환하고 모든 프로세스를 중단해야함.

Scenario 2: 연쇄 뱃지 생성 트랜잭션
- Given: 무사히 기한 내에 수락('accept') 버튼 조작
- When: 트랜잭션이 성공
- Then: 제안의 상태만 변경된 게 아니라, 연결된 `Badge` DB 레코드 1건이 동시에 Create 되어있어 즉각 SI의 신뢰도 등급(UI 노출권)을 보장해야 한다.

## :gear: Technical & Non-Functional Constraints
- 구조 제약: 수락 시 뱃지 생성 과정에서 오류가 나면 '제안 수락 상태'도 함께 원상 복귀(Rollback)되어야 하므로 `Promise.all` 이나 단일 `$transaction` 스코프 안에 이 모든 행위를 집어넣는 설계 컨벤션을 권장.
