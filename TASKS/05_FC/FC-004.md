## :white_check_mark: Task Breakdown (실행 계획)
- [ ] `otplib` 등 경량 TOTP 검증 라이브러리 추가 도입.
- [ ] (최초 로그인) Admin 계정에 TOTP Secret Key 생성 및 `qrcode` 라이브러리로 브라우저 화면에 뿌릴 URI 생성 API 제공 (`generateMfaSecret`).
- [ ] (로그인 검문소) FC-003 로그인 이후, 만약 권한이 Admin이라면 세션 상태를 `mfa_required` 로 부여.
- [ ] 2차 인증 액션 (`verifyMfaToken`): 유저가 입력한 6자리 코드를 DB상 저장된 Secret과 비교 후, 통과 시 세션을 최종 `authenticated` 및 권한 개방 상태로 전환.

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 로그인 방어 중간 검문소
- Given: Admin ID/Password가 뚫린 상태 (모의 공격 환경)
- When: 해커가 1차 로그인을 성공하여 JWT 토큰을 획득함
- Then: 시스템의 미들웨어 설정(FC-003)과 연계되어, 현재 세션이 `isMfaVerified = false` 라면 대시보드 데이터나 에스크로 송금 Action에 접근 시 무조건 403 Access Denied 혹은 `MFA 입력 페이지`로 리다이렉션 당해야 합니다.

Scenario 2: 코드 만료 룰
- Given: 어드민이 구글 OTP 앱에서 코드를 켰으나 시간을 초과함 (30~60초 룰 오프)
- When: 코드를 입력하여 검증(`verifyMfaToken`) 요청
- Then: 시간 오차 범위를 넘은 토큰은 백엔드단에서 `INVALID_TOTP_TOKEN` 에러로 단호히 차단된다.

## :gear: Technical & Non-Functional Constraints
- 보안/설계: TOTP Secret Key는 DB에 저장될 시 일반 Text가 아닌 대칭키 암호화 수준으로 말아 넣을지 프로젝트 암호화 정책(Symmetric Encryption)을 리뷰해야 함. (MVP 단계는 일반 저장도 허용, 단 주석 기재 필수).
