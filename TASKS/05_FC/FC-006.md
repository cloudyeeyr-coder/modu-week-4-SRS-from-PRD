## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 권한 가드 확인: 백엔드 모듈 레이어에서 `session.role === 'admin'` 이며, (FC-004) MFA 토큰이 인증되었는지를 한 번 더 체킹하는 엄격한 구조 마련.
- [ ] 선행 조건 검증: 입금 확인할 타겟 `Contract` 의 현재 상태가 무조건 `pending` 인지 확인 (이미 `escrow_held` 이면 중복 승인 에러 발생).
- [ ] 금액 검증: Admin이 입력한 `amountDeposited` 가 Contract의 `totalAmount` 와 일치하는지 백엔드 레벨 검사(`Amount Mismatch` 차단).
- [ ] 트랜잭션 실행

  - 2. `EscrowTx.state` -> `held` 로 전이
  - 3. (옵션) 알림 모듈(API-025 연계) 호출 지시하여 수요기업/SI에게 알림 발송 큐 적재.

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 중복 예치 승인 시도
- Given: 이미 상태가 `escrow_held` 인 계약건 데이터
- When: 어드민이 새로고침 실수나 다중 탭으로 승인 버튼을 두 번 누름
- Then: 백엔드의 상태 선행 조건 룰 엔진에 의해 즉각 반려(`ESCROW_ALREADY_HELD`)당하며 DB의 무결성이 손상되지 않는다.

Scenario 2: 금액 미스매치 감지
- Given: 총 계약 금액 1억원 중, 관리자가 5,000만원만 입력해서 Action을 호출
- When: 룰 검증 수행
- Then: 이 로직은 단호하게 400 Bad Request에 준하는 에러 객체(`ESCROW_AMOUNT_MISMATCH`)를 던져 시스템을 방어한다.
