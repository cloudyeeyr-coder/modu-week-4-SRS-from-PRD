## :dart: Summary
- 기능명: [API-021] RaaS 비교 결과 PDF (`POST /api/raas/pdf`) Route Handler DTO

## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 입력 `RaasPdfRequestSchema`: 프론트에서 계산된 결과 집합 또는 `modelCode`+`quantity` 패러미터를 다시 Body 인자로 맵핑 받을 Zod 스크립트 작성.
- [ ] 출력/반환 형태: `application/pdf` 컨텐트 타입 또는 JSON(`{ success, pdfUrl }`) DTO 명세 작성.
- [ ] 내용 명세: 이 DTO를 통해 전달된 데이터가 시각화 서버를 타서 'ROI 그래프', '월비용 테이블', 'TCO 비교' 등 3개 섹션의 필수 정보로 변환된다는 주석 포함.
- [ ] 에러 정의: `RAAS_PDF_GEN_FAILED`, `INVALID_PDF_PARAMS`

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: PDF 생성을 위한 페이로드 수신
- Given: 계산이 완료된 상태의 파라미터(`BasePrice` 등을 포함 혹은 조회 파라미터)
- When: 서버로 POST 요청을 보냄
- Then: 이 명세의 DTO는 해당 복합 객체를 Zod 구조체로 무사히 역직렬화하여 `jspdf`나 렌더 모듈로 던질 준비를 마쳐야 한다.

Scenario 2: 인터페이스 타입 약속
- Given: 프론트 개발자가 API 문서를 참고함
- When: 개발 진행 시
- Then: 반환 받을 데이터 형식이 순수 Buffer인지, File Blob URL인지 DTO 인터페이스 파일 내에 명확히 표기되어 불필요한 캐스팅 에러를 줄인다.

## :gear: Technical & Non-Functional Constraints
- 성능 검토: 3초 미만(REQ-FUNC-019)의 PDF 생성을 만족하기 위해, 본 REST API는 페이로드의 크기를 간소화하고, 필요 시 브라우저 측 Client Component에서 blob으로 넘겨 렌더하는 방식(프론트 로드)도 함께 의사결정 해둘 수 있도록 유도성을 부여.
