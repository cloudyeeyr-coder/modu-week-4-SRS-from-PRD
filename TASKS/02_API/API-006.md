## :dart: Summary
- 기능명: [API-006] 분쟁 접수 (`POST /api/escrow/dispute`) Route Handler DTO

## :white_check_mark: Task Breakdown (실행 계획)
- [ ] REST API URL 패턴 정의 및 허용 Method(POST) 설계 명세화
- [ ] Request Body(JSON) Zod 체커 설계: `contractId`, `raisedBy` (buyer | si_partner), `reason` (텍스트)
- [ ] Response Body 구조: `status: success`, 변경된 Contract의 상태값(`disputed`), 그리고 어드민/고객센터 접수번호
- [ ] 에러 코드 목록 작성 (`DISPUTE_INVALID_STATE`, `DISPUTE_EMPTY_REASON`)

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 올바른 JSON 파싱 검증
- Given: `{ "contractId": "x", "raisedBy": "buyer", "reason": "품질 불량" }`
- When: Fetch 등의 클라이언트가 요청함
- Then: Route Handler 내 DTO 레벨의 파싱 모듈이 400 에러를 던지지 않고 무사히 Model Payload로 캐스팅하여 통과시킨다.

Scenario 2: 필수 필드 누락 검증
- Given: 사용자가 사유(`reason`)를 적지 않고 POST함
- When: 파싱 파이프라인 작동
- Then: 규격 상 `DISPUTE_EMPTY_REASON` 또는 Zod의 400 Bad Request 포맷이 발동되어 이를 프론트엔드로 리턴한다.

## :gear: Technical & Non-Functional Constraints
- 구조 제약: Server Action 대신 범용적인 Route Handler로 구축된 API이므로(SRS 6.1 규격 참조), 반환 폼이 Server Action 폼(`{ success, errors }`)보다는 `HTTP Status Code` 및 `{ error: string }` 기반의 순수 REST 포맷을 준수해야 함.
