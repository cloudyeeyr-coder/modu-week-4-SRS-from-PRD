## :white_check_mark: Task Breakdown (실행 계획)
- [ ] DTO Zod 인터페이스 정의 (`InspectionPayloadSchema`)
  - 필드: `contractId`, `decision` (`approve` | `reject`), `rejectReason` (선택적 사항, 단 `reject` 시 필수 여부 로직 등)
- [ ] (규칙) `approve` 시 상태는 `release_pending` 으로 전이됨을 문서 명세로 확약.
- [ ] (규칙) `reject` 시 상태는 `disputed` 로 전이되며 관련된 제약 정의.
- [ ] Response Payload: 처리 결과(`success`) 및 변경된 Contract 상태
- [ ] 에러 코드 목록 작성 (`INVALID_INSPECTION_DECISION`, `EMPTY_REJECT_REASON`)

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 거절 폼 유효성
- Given: 사용자가 Decision을 `reject` 로 선택
- When: `rejectReason` (거절 사유) 칸을 빈칸으로 두고 폼을 서브밋(호출) 시도함
- Then: Zod 스키마 등의 조건부 검증(Conditional Validation)이 동작하여 "거절 시 사유는 필수입니다" 라는 `EMPTY_REJECT_REASON` 에러 메시지를 포함해 거부한다.

Scenario 2: 승인 및 성공 결과
- Given: 수요기업이 정상적으로 `approve` 처리를 결정함
- When: 액션이 호출되어 수행됨(모의)
- Then: `{ success: true, contractState: 'release_pending' }` 형태로 리턴하여 프론트엔드가 이를 기반으로 화면을 낙관적 렌더링(Optimistic update) 가능하게 한다.
