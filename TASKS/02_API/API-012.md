## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 백엔드 Route URL `POST /api/warranty/issue` 설계
- [ ] Request Body: `contractId` 혹은 `escrowTxId` (어느 것을 기준으로 할지 명확화)
- [ ] Response 헤더/구조: PDF 생성이 동기로 일어나는 경우 `Content-Type: application/pdf` 의 바이너리 반환 규격이거나, 비동기 생성 후 DB에 박힌 URL 객체 `{ url: "...", success }` 를 리턴할지 구조 결정(MVP 상은 URL JSON 리턴 권장).
- [ ] 에러 코드 목록 작성 (`WARRANTY_ISSUE_FAILED`, `CONTRACT_NOT_ELIGIBLE`)

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 미자격 계약 거부
- Given: 계약이 진행 중(inspecting)인 상황의 파라미터(`contractId`)
- When: 보증서 발급 엔드포인트를 때림
- Then: "에스크로 결제가 완료(released)되어야 발급할 수 있습니다"를 뜻하는 에러 코드 명세가 반환되어야 한다.

Scenario 2: 인터페이스 바이너리 vs URL 타입 규칙 확인
- Given: 올바른 발행 요청
- When: 성공 응답
- Then: 이 DTO 명세에 작성된 대로, JSON으로 래핑된 URL 형태인지 순수 Buffer/Stream 형태인지 타겟 프론트엔드/호출자가 예측 및 파싱 가능하다.
