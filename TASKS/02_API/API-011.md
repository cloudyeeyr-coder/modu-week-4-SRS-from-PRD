## :white_check_mark: Task Breakdown (실행 계획)
- [ ] 입력 Request DTO 설계: `ticketId`, `resolutionDetail` (완료 상세 코멘트)

- [ ] 에러 규격 설계 (`TICKET_NOT_FOUND`, `TICKET_ALREADY_RESOLVED`)

## :test_tube: Acceptance Criteria (BDD/GWT)
Scenario 1: 올바른 DTO 및 SLA 충족
- Given: 올바른 `ticketId`가 포함된 액션 호출
- When: 서버 액션이 리턴할 때
- Then: 반환된 결과로 `slaMet` 이 `true`나 `false` 형식으로 문제 없이 응답 포맷에 포함되어 내려가야 한다.

Scenario 2: 상태 위반 방어
- Given: 이미 완료된(`resolvedAt` 필드가 찬) 티켓
- When: 다시 `resolveTicket`을 호출하려 함
- Then: DTO 규격서에 명시된 에러 코드(`TICKET_ALREADY_RESOLVED`)가 반환되어 이중 보고를 방지해야 한다.

## :gear: Technical & Non-Functional Constraints
- 구조 제약: SLA 판정 로직 자체는 DTO 단이 아니라 Service나 Action 레벨에서 `Date.now()` 기준으로 처리될 것이므로, 클라이언트가 파라미터로 "나 24시간 내에 완료했어요" 같은 플래그를 던지지 못하도록 인터페이스를 엄격히 Close 시킨다.
